HTTP

HTTP stands for Hyper Text Transfer Protocol. A protocol is a set of formal rules describing how to transmit or exchange data, especially across a network.

When you request a webpage via your internet browser using a URL e.g. http://humanevolution.com/simulations.html, various steps occur, these include:

  1. Requesting the IP address for the server, where your website is stored
  2. Now you know where the website is stored, sending an HTTP request to the server for the particular page you'd like to see
  3. The webframework e.g Express will then need to convert your http request into an object with request and response parameters
  4. Next, this object will be passed into one or more handler functions set up on the server in Express. These will be executed if their path matches the path/endpoint (e.g. page) that you requested. These functions could do various things such as send back a static HTML page. They might also fetch content from a database, and put it into a template ready to be sent back.
  5. The resulting object will finally need to be converted into HTTP so that the requested information can be sent back to your browser.

    6. These steps are explained in more detail below.

HTTP Requests
  1. First, the browser needs to get the IP address of the server where the requested webpage i.e. www.humanevolution.com/simulations.html is stored.

    2. To do this, it extracts the domain name from the url i.e. humanevolution.com and asks the DNS for the IP address. If the DNS server has the address for the server where that domain is stored, it will return it. Otherwise it will forward the query along to the DNS server it is configured to defer to. The DNS server will then send the IP address back to the client. The client machine now has an IP address that uniquely identifies a machine on the internet.

  2. Next the client assembles an HTTP request. This has a HEADER consisting of various parts including:
    • The specific path requested, in this case /simulations.html
    • The HTTP version being used e.g. HTTP 1.1
    • The user-agent. This might be curl if you're using your terminal to send the request and will be displayed as a key:value pair e.g. User-Agent: curl/7.37.1
    • Host: e.g. humanevolution.com
  3. Once the request reaches the server, it will be converted into an object. What happens after this will be explained in more detail here.
  4. Once the HTTP request has been processed by the server, it will convert the resulting object into HTTP response to send back to the browser. The response will include various HEADER fields including:
    • The version of the HTTP protocol e.g HTTP 1.1
    • The Status code e.g. 200 which means that the page has been processed and return successfully. See here for further types of HTTP status codes.
    • the server that is running e.g. nginx/1.6.2
    • content-type: text/html; charset = utf-8
    • content-length: 173747
    • set-cookie: vid=196338346 (This is how a cookie gets set)
    • i.e. the actual docs being sent over
  5. The transaction is now over. if you start again, it doesn't remember who you are. This is a concept called being stateless
  6. See for a much more thorough list of header fields.

HTTP Status Codes

Status codes fall into five main categories, as outline below. For a full list see Wikipedia:

  1. 1XX INFORMATIONAL
    • 100: Continue
    • 101: Switching Protocols
  2. 2XX SUCCESS
    • 200: OK - Standard response for HTTP requests
    • ....
  3. 3XX REDIRECTION
    • 301: Move permanently: This and future requests should be directe to the given URL
    • ....
  4. 4XX CLIENT ERROR
    • Bad request
    • Unauthorised
    • 418: I'm a teapot
  5. 5XX SERVER ERROR
    6. The server failed to fulfil an apparently valid request.
    • 500: Internal server error

Dev Tools

Various Dev tools exist for inspecting request headers. One web too is httpbin.org You can also use the httpfox extension in Firefox or Fiddler for Windows.

In Chrome you can use Dev tools to inspect the request and it's headers:

  • Visit the URL
  • Right-click - select inspect
  • Select the Network tab
  • Reload the page - select the URL of interest - HTTP Headers are displayed on the right-hand panel

Request Methods

HTTP request methods indicate the action that should be performed on the identified resource. See Wikipedia for more details.

Some of the main request methods are:

  • GET: The GET method requests a representation of the specified resource. Requests using GET should only retrieve data and should have no other effect.
  • HEAD: The HEAD method asks for a response identical to that of a GET request, but without the response body. This is useful for retrieving meta-information written in response headers, without having to transport the entire content.
  • POST: The POST method requests that the server accept the entity enclosed in the request as a new subordinate of the web resource identified by the URI. The data POSTed might be, for example, an annotation for existing resources; a message for a bulletin board, newsgroup, mailing list, or comment thread; a block of data that is the result of submitting a web form to a data-handling process; or an item to add to a database.
  • PUT: The PUT method requests that the enclosed entity be stored under the supplied URI. If the URI refers to an already existing resource, it is modified; if the URI does not point to an existing resource, then the server can create the resource with that URI.
  • DELETE: The DELETE method deletes the specified resource.

Abstractions

Node.js has a library called HTTP - thsi gets back an HTTP request. For Python there is Flask. Ember is another high level abstraction. AngularJS is similar to Node, but in this case headers will come back in a JSON object.
